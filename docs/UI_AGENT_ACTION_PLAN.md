# nopolicybot — план действий агента: UI

## Назначение файла

Операционный план для AI-агента по улучшению UI Modern Dashboard на основе визуального аудита интерфейса. Каждый этап содержит конкретные действия, критерии завершения и точки проверки. БД-related задачи в этом файле не рассматриваются.

---

## Соглашения

- `[ПРОВЕРИТЬ]` — агент читает текущий код/состояние перед действием
- `[ВЫПОЛНИТЬ]` — конкретное действие которое нужно реализовать
- `[ТЕСТ]` — проверка что действие выполнено корректно
- `[GATE]` — блокирующая проверка; следующий этап не начинается до прохождения

---

## Этап UI-1: Статус WebSocket — исправить цвет и текст

- `[ПРОВЕРИТЬ]` Найти в коде место где рендерится `Live: ws reconnecting...` — цвет и логика состояний
- `[ВЫПОЛНИТЬ]` Заменить цветовую схему статусов:
  - `connected` → зелёный
  - `reconnecting` → оранжевый + анимированная точка
  - `disconnected` → красный
- `[ВЫПОЛНИТЬ]` Текст `ws reconnecting...` заменить на человекочитаемый: «Соединение восстанавливается...»
- `[ТЕСТ]` Симулировать обрыв WS — убедиться что цвет меняется корректно
- `[GATE]` Зелёный цвет присваивается только при реальном `connected` состоянии

## Этап UI-2: Опасные кнопки в Ops Controls — защита от случайного нажатия

- `[ПРОВЕРИТЬ]` Найти обработчики кнопок «Перезапустить бота» и «Сбросить полит. счётчик»
- `[ВЫПОЛНИТЬ]` Добавить confirm-диалог перед выполнением каждого деструктивного действия: «Вы уверены? Это действие нельзя отменить»
- `[ВЫПОЛНИТЬ]` Визуально отделить опасные кнопки от нейтральных: outline красного цвета для «Перезапустить бота» и «Сбросить полит. счётчик»
- `[ВЫПОЛНИТЬ]` После выполнения — показывать toast-уведомление с результатом (успех / ошибка)
- `[ТЕСТ]` Нажать опасную кнопку — убедиться что диалог появляется и действие не выполняется без подтверждения
- `[GATE]` Ни одно деструктивное действие не выполняется без явного подтверждения

## Этап UI-3: Единый язык интерфейса

- `[ПРОВЕРИТЬ]` Пройти по всем блокам страницы, составить список английских заголовков и подписей
- `[ВЫПОЛНИТЬ]` Перевести все заголовки блоков на русский:
  - `Ops Controls` → «Управление»
  - `Graph Lab` → «Граф сообщества»
  - `Chat Digest & Analysis Preview` → «Дайджест и анализ»
  - `Storage Status` → «Статус хранилища»
  - `Recommendations + Predictive` → «Рекомендации»
  - `Chat Health Overview` → «Здоровье чата»
  - `Community Structure` → «Структура сообщества»
  - `User Leaderboard` → «Рейтинг участников»
  - `Advanced Analytics Panel` → «Аналитика»
  - `Quick View` / `Advanced Lab` → «Обзор» / «Детальный режим»
- `[ТЕСТ]` Пройти по странице — убедиться что английских заголовков не осталось
- `[GATE]` Весь интерфейс на одном языке

## Этап UI-4: Блок «Ранги» — выделить аномалию Unknown

- `[ПРОВЕРИТЬ]` Найти рендеринг блока «Ранги», логику подсчёта Unknown
- `[ВЫПОЛНИТЬ]` Добавить визуальный индикатор: если Unknown > 50% от общего числа пользователей — показывать предупреждение под списком: «85% пользователей не классифицированы — запустите анализ»
- `[ВЫПОЛНИТЬ]` Добавить кнопку прямо в блок: «Классифицировать» → запускает портрет для всех Unknown пользователей
- `[ВЫПОЛНИТЬ]` Строку Unknown выделять жёлтым если > 50%, красным если > 80%
- `[ТЕСТ]` При Unknown > 50% — убедиться что предупреждение появляется
- `[GATE]` Аномальное состояние классификации визуально очевидно без анализа цифр

## Этап UI-5: Граф — интерактивность и layout

- `[ПРОВЕРИТЬ]` Изучить текущую библиотеку рендеринга графа и параметры force layout
- `[ВЫПОЛНИТЬ]` Настроить force-directed параметры:
  - Увеличить `repulsion` / `charge` чтобы узлы расталкивались и не собирались в шар
  - Уменьшить `linkDistance` для сильно связанных узлов, увеличить для слабо связанных
- `[ВЫПОЛНИТЬ]` Убрать постоянные подписи с узлов — показывать имя только при hover
- `[ВЫПОЛНИТЬ]` Добавить обработчик клика на узел → боковая панель справа с карточкой:
  - Имя / username
  - Ранг и influence score
  - Количество связей
  - Последняя активность
  - Кнопка «Открыть профиль»
- `[ВЫПОЛНИТЬ]` При клике на узел — подсветить его связи, затемнить остальные
- `[ТЕСТ]` Кликнуть на узел — убедиться что панель открывается с правильными данными
- `[ТЕСТ]` Граф с 30 узлами — убедиться что узлы не образуют плотный шар
- `[GATE]` Граф читаем, клик на узел работает, подписи не перекрывают рёбра

## Этап UI-6: Recommendations — карточки вместо списка

- `[ПРОВЕРИТЬ]` Изучить API endpoint который отдаёт рекомендации — структуру данных
- `[ВЫПОЛНИТЬ]` Заменить список `[medium] retention_standard 359858187: Проверить вовлечённость` на карточки:
  ```
  [Имя пользователя]             [medium]
  Снижение вовлечённости
  Рекомендация: дать точку входа в обсуждение
  [Открыть профиль]  [Отметить выполненным]
  ```
- `[ВЫПОЛНИТЬ]` Числовые ID пользователей заменить на имена (join с user data)
- `[ВЫПОЛНИТЬ]` Приоритет `[medium]` / `[high]` / `[low]` отображать цветным бейджем, не текстом в скобках
- `[ВЫПОЛНИТЬ]` Кнопка «Отметить выполненным» — скрывает карточку и пишет в audit log
- `[ТЕСТ]` Убедиться что имена подтягиваются для всех рекомендаций
- `[GATE]` Ни одного числового ID в списке рекомендаций, каждая карточка имеет действие

## Этап UI-7: Predictive метрики — визуализация вместо цифр

- `[ПРОВЕРИТЬ]` Найти рендеринг блоков `churn_risk`, `toxicity`, `virality`
- `[ВЫПОЛНИТЬ]` Заменить текстовую строку `current: 0.455 · predicted: 0.455 · delta: 0.000 · conf: 0.33` на визуальный компонент:
  - Прогресс-бар 0→1 с текущим значением
  - Стрелка тренда: ↑ если predicted > current, ↓ если меньше, → если delta < 0.01
  - Confidence отображать как подпись под баром: «Уверенность: 33%»
- `[ВЫПОЛНИТЬ]` Цветовая схема:
  - `churn_risk` и `toxicity` — красный при > 0.6, жёлтый при 0.3–0.6, зелёный при < 0.3
  - `virality` — инвертировано (высокий = хорошо): зелёный при > 0.6, жёлтый при 0.3–0.6, красный при < 0.3
- `[ТЕСТ]` Убедиться что цвета корректно применяются при разных значениях
- `[GATE]` Метрики читаются без знания предметной области — понятно «хорошо» или «плохо»

## Этап UI-8: Chat Health — расшифровать числа

- `[ПРОВЕРИТЬ]` Найти рендеринг блока Chat Health Overview, выяснить что означает число в скобках `(0.400)` — целевое значение, историческое или максимум
- `[ВЫПОЛНИТЬ]` Добавить tooltip или подпись к каждому числу:
  - `Health: 0.238 ↗ (0.400)` → подпись: «Текущее (целевое)» или «Текущее (7 дней назад)» — в зависимости от семантики
  - Аналогично для Engagement, Retention
- `[ВЫПОЛНИТЬ]` Добавить цветовую индикацию: значение < 50% от целевого — красный, 50–80% — жёлтый, > 80% — зелёный
- `[ВЫПОЛНИТЬ]` Topics отображать как кликабельные теги, а не plain text через точку с запятой
- `[ТЕСТ]` Навести на число — убедиться что tooltip появляется с объяснением
- `[GATE]` Любой новый пользователь понимает что означает каждое число без документации

## Этап UI-9: At-risk список — дифференциация

- `[ПРОВЕРИТЬ]` Изучить API at-risk endpoint — какие данные реально возвращаются кроме risk score и текста рекомендации
- `[ВЫПОЛНИТЬ]` Найти причину почему все пользователи получают одинаковый текст `Soft check-in and topic re-entry` — исправить алгоритм чтобы тип причины дифференцировался:
  - `activity_drop` → «Резкое снижение активности»
  - `conflict_rise` → «Рост конфликтных сообщений»
  - `isolation` → «Изоляция в графе сообщества»
  - `retention_risk` → «Риск оттока`
- `[ВЫПОЛНИТЬ]` Каждый пользователь в at-risk списке должен иметь уникальную причину риска
- `[ВЫПОЛНИТЬ]` Добавить сортировку по risk score (после дифференциации значения перестанут быть одинаковыми)
- `[ВЫПОЛНИТЬ]` Добавить быстрое действие прямо в строке: «Написать» (DM через бота) / «Снять флаг»
- `[ТЕСТ]` Убедиться что разные пользователи получают разные причины риска
- `[GATE]` Список at-risk дифференцирован, каждая запись имеет кнопку действия

---

## Этап UI-10: Leaderboard — динамика

- `[ПРОВЕРИТЬ]` Найти данные leaderboard — есть ли исторические score для сравнения с предыдущим периодом
- `[ВЫПОЛНИТЬ]` Добавить колонку тренда рядом со score: ↑ / ↓ / → на основе сравнения с предыдущим периодом
- `[ВЫПОЛНИТЬ]` Если два пользователя имеют одинаковый score (например, оба 0.800) — добавить вторичную сортировку по активности за последние 24h
- `[ВЫПОЛНИТЬ]` Имена сделать кликабельными → открывать профиль пользователя
- `[ТЕСТ]` Убедиться что клик по имени открывает профиль
- `[GATE]` Leaderboard показывает динамику, нет визуально одинаковых строк подряд

## Контрольная проверка после всех этапов

- Пройти по странице — ни одного английского заголовка
- Нажать каждую опасную кнопку — диалог подтверждения появляется
- WS статус — зелёный только при активном соединении
- Граф — клик на узел открывает карточку
- At-risk список — у каждого пользователя своя причина риска
- Рекомендации — ни одного числового ID, у каждой карточки есть кнопка действия
- Leaderboard — присутствует колонка тренда
